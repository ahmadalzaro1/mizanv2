# Phase 09.1: Bias Auditor Rework — Research

**Researched:** 2026-03-03
**Domain:** ML fairness auditing, NLP bias evaluation, D3.js visualization, FastAPI progress streaming
**Confidence:** MEDIUM-HIGH (stack verified; Arabic NLP fairness is a thin ecosystem)

---

## Summary

The existing Bias Auditor is a minimum viable implementation: it runs MARBERT on 560 examples and
returns per-category F1/precision/recall in a horizontal bar chart. For an NLP researcher like Lina,
this tells the *what* but not the *why*. The field of ML bias auditing has established a standard
conceptual framework (Aequitas, 2018–2024) around **group metrics** (absolute per-subgroup
performance) and **disparity metrics** (relative ratios between subgroups), plus the importance of
**false positive rate parity** as a distinct fairness concept from recall. The current implementation
misses false positive rate per category entirely because it only tracks FP in the overall matrix.

The most impactful rework additions for a 5-minute hackathon pitch are: (1) **confidence
distribution histograms per category** — showing whether the model is uncertain or overconfident on
specific hate types, (2) **per-dataset-source breakdown** — the 4 source datasets (JHSC, OSACT5,
L-HSAB, Let-Mi) have different dialects and annotation schemes, so model performance per source
reveals domain adaptation failures, and (3) **false positive sample display** — showing which
non-hate examples are being flagged as hate, giving Lina actionable insight into what's going wrong.

A real-time progress indicator during the 140s audit run replaces the current silent wait, using
FastAPI Server-Sent Events (SSE). This is the established pattern for long-running ML tasks in
FastAPI and directly addresses the worst UX problem of the current implementation.

**Primary recommendation:** Add confidence distributions + per-source breakdown + FP sample viewer
to the existing backend, surface with new D3 histogram chart and tabbed layout on frontend. Use
FastAPI SSE for progress. Do NOT add Aequitas (Python library dependency cost not worth it for what
we already have). Do NOT add intersectional analysis (requires 2x2 cross-category data that doesn't
exist at this sample size).

### Pattern 1: FastAPI SSE for Long-Running Progress

**What:** Stream progress events from the 560-example inference loop to the frontend.
**When to use:** Any blocking operation > 5 seconds where UX feedback matters.

```python
# Source: FastAPI official docs + community pattern (HIGH confidence)
from fastapi.responses import StreamingResponse
import asyncio, json

@router.post("/run/stream")
async def run_audit_stream(
    current_user: Annotated[User, Depends(get_current_user)],
    db: Session = Depends(get_db),
):
    """Stream audit progress via SSE."""
    async def event_generator():
        examples = db.query(ContentExample).all()
        total = len(examples)
        for i, ex in enumerate(examples):
            # Run inference (blocking — wrap in executor for async)
            result = await asyncio.get_event_loop().run_in_executor(
                None, model_manager.classify, ex.text, settings.code_mixed_threshold
            )
            progress = round((i + 1) / total * 100)
            yield f"data: {json.dumps({'progress': progress, 'current': i+1, 'total': total})}\n\n"
        # Final event with full results
        yield f"data: {json.dumps({'done': True, 'results': results})}\n\n"

    return StreamingResponse(event_generator(), media_type="text/event-stream")
```

**Frontend EventSource consumption:**
```typescript
// Source: MDN EventSource API + community pattern (HIGH confidence)
const source = new EventSource(`${API_URL}/api/audit/run/stream`, {
  // EventSource doesn't support custom headers — must pass token via query param or cookie
  // For JWT: use a short-lived token URL param, or fall back to regular POST + polling
})
source.onmessage = (event) => {
  const data = JSON.parse(event.data)
  if (data.done) { setAuditRun(data.results); source.close() }
  else { setProgress(data.progress) }
}
```

**IMPORTANT CAVEAT:** EventSource does not support custom request headers. Since the backend uses
JWT Bearer auth, SSE requires either: (a) query-param token (security tradeoff), or (b) use
`fetch()` with `ReadableStream` instead of `EventSource`. The fetch approach works with headers.

```typescript
// Preferred pattern for JWT + SSE (MEDIUM confidence — verified pattern)
const response = await fetch(`${API_URL}/api/audit/run/stream`, {
  method: 'POST',
  headers: { Authorization: `Bearer ${token}` },
})
const reader = response.body!.getReader()
const decoder = new TextDecoder()
while (true) {
  const { done, value } = await reader.read()
  if (done) break
  const lines = decoder.decode(value).split('\n')
  for (const line of lines) {
    if (line.startsWith('data: ')) {
      const data = JSON.parse(line.slice(6))
      if (data.progress !== undefined) setProgress(data.progress)
      if (data.done) { setAuditRun(data.results); break }
    }
  }
}
```

### Pattern 2: D3.js Confidence Histogram

**What:** Show the distribution of model confidence scores for each hate category.
**When to use:** Diagnosing whether a model is uncertain (bad signal) or overconfident (calibration issue).

```typescript
// Source: Context7 /d3/d3 — d3.bin() API (HIGH confidence)
import * as d3 from 'd3'

// Inside useEffect:
const bin = d3.bin()
  .value((d: number) => d)
  .domain([0, 1])
  .thresholds(10)  // 10 buckets: 0-10%, 10-20%, ... 90-100%

const bins = bin(confidenceScores)  // confidenceScores: number[]

// Each bin: { x0: number, x1: number, length: number }
const yScale = d3.scaleLinear()
  .domain([0, d3.max(bins, b => b.length) ?? 0])
  .range([height, 0])

const xScale = d3.scaleLinear()
  .domain([0, 1])
  .range([0, width])

// Render bars
svg.selectAll('rect')
  .data(bins)
  .join('rect')
  .attr('x', b => xScale(b.x0))
  .attr('y', b => yScale(b.length))
  .attr('width', b => Math.max(0, xScale(b.x1) - xScale(b.x0) - 1))
  .attr('height', b => height - yScale(b.length))
  .attr('fill', '#1e3a5f')
```

### Anti-Patterns to Avoid

- **Re-running all 560 examples every time for new views:** Compute all metrics in a single pass;
  store in JSONB. Do not add separate endpoints that re-run inference.
- **Using Aequitas Python library:** It adds pandas + heavy dependencies for metrics we can compute
  manually in 20 lines. Not worth the dependency cost.
- **Intersectional analysis (gender × religion):** At 560 examples, cross-tabulated cells have < 5
  samples each. Metrics on < 5 samples are statistically meaningless and misleading.
- **Threshold sensitivity / ROC curves:** Requires raw probability arrays for 560 examples stored
  outside JSONB. High engineering cost, low demo value for a 5-minute pitch.
- **Custom test set upload:** Requires file upload endpoint, parsing, MARBERT batch inference on
  arbitrary input — out of scope for a hackathon polish phase.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Confidence histogram bucketing | Custom loop to count ranges | `d3.bin()` with thresholds | D3's bin handles edge cases (inclusive x1, empty bins) correctly |
| SSE event parsing | Custom stream parser | Standard `data: ...\n\n` SSE format + `fetch` ReadableStream | EventSource spec is well-defined; don't invent a custom format |
| Per-source metrics aggregation | Separate API endpoint + extra inference | Extend the single audit loop | Inference is the bottleneck; do one pass, capture all dimensions |
| Arabic number display in histograms | Custom JS formatting | Existing `toArabicDigits` from `format.ts` | Already exists and tested in the project |
| False positive identification | Post-processing pass after audit | Inline flag during inference loop | Avoids second pass through examples |

**Key insight:** Everything new in this phase is computed in the *same 560-example inference loop*.
The results JSONB grows from ~10KB to ~25KB — still well within PostgreSQL JSONB performance range.

## Common Pitfalls

### Pitfall 1: EventSource Cannot Send JWT Headers

**What goes wrong:** Developer uses `new EventSource('/api/audit/run/stream')` — it ignores the
`Authorization: Bearer` header, returning 401 silently or falling back to an expired session.
**Why it happens:** The W3C EventSource spec only supports GET requests with no custom headers.
**How to avoid:** Use `fetch()` + `ReadableStream` (see Pattern 1 above). Pass the token from
`localStorage.getItem('mizan_token')` in the Authorization header.
**Warning signs:** 401 in the network tab while EventSource appears connected.

### Pitfall 2: SQLAlchemy Session Closed Mid-SSE

**What goes wrong:** The SSE generator is async; the SQLAlchemy session created by `get_db()`
dependency may close before the generator finishes yielding.
**Why it happens:** FastAPI closes the DB session after the route function returns. With a generator,
the function "returns" when it yields the first value.
**How to avoid:** Load all examples into memory before starting the generator:
```python
examples = db.query(ContentExample).all()  # Load eagerly, before first yield
db.close()  # Explicitly close; don't rely on context manager
```

### Pitfall 3: JSONB Schema Change Breaks Cached Results

**What goes wrong:** Old `BiasAuditRun` rows in the DB have `results` without `per_source` or
`confidence_dist` keys. The frontend crashes trying to access `results.per_source`.
**Why it happens:** JSONB is schemaless — old runs return old structure.
**How to avoid:** Frontend must treat `per_source`, `confidence_dist`, `false_positives` as optional
with fallback to empty state. Never assume new keys exist in cached results.
```typescript
const perSource = auditRun?.results?.per_source ?? []
const confidenceDist = auditRun?.results?.confidence_dist ?? {}
const falsePosiitveSamples = auditRun?.results?.false_positives ?? []
```

### Pitfall 4: Per-Category False Positive Rate is Undefined

**What goes wrong:** The current `category_stats` dict only tracks hate examples. FP rate requires
tracking non-hate examples predicted as hate — which don't have a category in ground truth.
**Why it happens:** FP analysis is cross-class by nature: a non-hate example has no hate_type, so
there's no "category" to assign the false positive to.
**How to avoid:** False positive rate per category is not meaningful in the current data model.
Track FPs by `source_dataset` instead (which all examples have). Per-category false positive
tracking would require a different data model (negative examples tagged with target categories).

### Pitfall 5: Confidence Histogram Looks Flat for Small Categories

**What goes wrong:** Categories with < 20 examples show a near-flat histogram that communicates
nothing — worse than not showing it.
**Why it happens:** With 10 bins across N < 20 examples, most bins have 0 or 1 items.
**How to avoid:** Only render the histogram for categories with `sample_count >= 20`. For smaller
categories, show a text note: "عدد العينات غير كافٍ للتوزيع (N=5)".

### Pitfall 6: 140-Second Audit + Progress Bar Stalls on Error

**What goes wrong:** An exception mid-loop kills the SSE stream without sending an error event.
The frontend progress bar stalls at e.g. 47%.
**Why it happens:** Python exceptions in a generator are swallowed silently.
**How to avoid:** Wrap the inference loop in try/except; yield an error event on failure:
```python
except Exception as e:
    yield f"data: {json.dumps({'error': str(e)})}\n\n"
    return
```

### Pitfall 7: BiasChart RTL Axis Label Clipping

**What goes wrong:** D3 renders Arabic text on the Y-axis left-anchored; RTL layout flips the
container direction, causing labels to clip outside the SVG margin.
**Why it happens:** D3 SVG rendering is not aware of CSS `dir="rtl"` on the parent container.
**How to avoid:** Use explicit `text-anchor: 'end'` with a fixed left margin (already in current
BiasChart). For new charts, mirror this pattern. Do not rely on CSS direction inheritance inside SVG.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Per-class accuracy only | F1/precision/recall per group + disparity ratios | Aequitas 2018, standard by 2022 | Current implementation is already aligned — missing disparity ratios |
| Single dataset evaluation | Multi-source / domain slicing | 2020–2023 | "Dataset slicing" is now expected in NLP papers; our per-source breakdown addresses this |
| Binary hate/not-hate fairness | Category-level fairness for fine-grained hate types | 2021–2024 Arabic NLP papers | Current implementation partially addresses — F1 per category exists |
| Silent batch processing | Real-time progress streaming | Standard UX expectation since 2020 | Our 140s silent wait is the worst UX problem; SSE progress directly fixes it |
| Aggregate confidence reporting | Per-class calibration/confidence distribution | Calibration research surge 2022–2024 | Histogram view is a direct implementation of this research trend |

**Deprecated/outdated in this domain:**
- Accuracy as sole metric: Replaced by F1 for imbalanced datasets. Current implementation already uses F1.
- Macro-averaged F1 only: Micro-averaged + per-class F1 are now expected. Current implementation reports per-class.
- Cohen's Kappa for agreement: Still used in annotation, but not in model fairness auditing.

**What's SOTA vs what training thinks:**
- Training: Assumes fairness toolkit = Aequitas or IBM AI Fairness 360 (AIF360). Reality: These are great for tabular/classification tasks, but for NLP hate speech at 560 examples, hand-rolled metrics are cleaner and more appropriate.
- Training: Assumes intersectional bias analysis is standard. Reality: Requires much larger datasets (thousands of examples per intersection). Not feasible here.
- Training: May suggest threshold optimization / ROC curves. Reality: For a binary hate classifier with fixed softmax output, threshold analysis is valid but requires storing raw probabilities per example — significant backend change for marginal demo value.

## What NOT to Build (Final Summary)

These were considered and rejected:

1. **Aequitas Python integration** — Adds pandas dependency, complex group/bias object model, and
   doesn't understand Arabic text categories. Manual metrics are simpler and equivalent.

2. **Intersectional bias** (gender × religion, dialect × category) — Statistically invalid at N=560
   with 9 categories × 2 dialects = ~31 examples per cell on average.

3. **Threshold sensitivity analysis** — Requires persisting raw logits/probabilities per example.
   Significant schema change + compute cost. Low demo value.

4. **Custom test set upload** — File parsing, validation, MARBERT batch on arbitrary input. Out of
   scope for a hackathon polish phase.

5. **Multi-run comparison / temporal progression** — Needs multiple cached runs with versioning.
   Not meaningful until the model changes between runs.

6. **Model-vs-model comparison (MARBERT vs XLM-R)** — Would require running both models on all
   560 examples (280s+). High compute cost, complex result display.

7. **PDF report generation** — Requires headless Chrome or reportlab. High dependency cost, low
   demo value beyond CSV download.

## Open Questions

1. **SSE vs. polling for the progress indicator**
   - What we know: SSE with fetch ReadableStream works with JWT Bearer headers. EventSource does not.
   - What's unclear: Whether the demo environment (Docker Compose) handles long-lived HTTP connections without timeout.
   - Recommendation: Implement SSE with fetch ReadableStream. Add 5-minute server timeout. If
     connection issues arise in demo, fall back to a 140s client-side timer showing estimated progress.

2. **Confidence score availability from current MARBERT checkpoint**
   - What we know: `model_manager.classify()` already returns `result["confidence"]` — verified
     in `ml_models.py`. The BiasChart already uses this signal indirectly.
   - What's unclear: Whether the confidence values from softmax are well-calibrated (Platt scaling
     not applied). For a demo, raw softmax probabilities are acceptable.
   - Recommendation: Use raw confidence; add a footnote "الثقة: احتمالية softmax غير معايَرة".

3. **False positive text display — privacy/content concerns**
   - What we know: The seeded content_examples include real hate speech tweets from JHSC/OSACT5.
     False positive examples are non-hate examples.
   - What's unclear: Whether showing actual tweet text in the UI is appropriate for the demo context.
   - Recommendation: Show the text. It's non-hate content (that's why it's a false positive).
     Truncate to 120 characters.
